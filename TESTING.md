# Testing

This SDK ships a test suite that runs against the **E2E sandbox**
(`api.sandbox.e2e.gr4vy.app`). It covers the full merchant transaction lifecycle
and the back-office configuration surface, and reaches every API operation.

> **Note for contributors:** this repository is generated by
> [Speakeasy](https://www.speakeasy.com/). Everything under `src/Gr4vy.Tests/`,
> `scripts/`, and this file is **not** regenerated and is safe to edit. The test
> project (`src/Gr4vy.Tests/Gr4vy.Tests.csproj`) is hand-maintained — add test-only
> NuGet packages there (e.g. `FsCheck`), not to the generated `src/Gr4vy` project.

## Prerequisites

- The [.NET 8 SDK](https://dotnet.microsoft.com/download).
- The signing key for the E2E sandbox. Provide it in either way:
  - Export it: `export PRIVATE_KEY="$(cat /path/to/private_key.pem)"`, **or**
  - Save it as `private_key.pem` in the repo root (git-ignored).

`src/Gr4vy.Tests/Utils/TestEnvironment.cs` reads `PRIVATE_KEY` first, then falls
back to `private_key.pem`.

## Running the tests

```sh
# Everything (offline unit tests + live E2E)
dotnet test src/Gr4vy.Tests

# A single suite (by namespace) or one fixture
dotnet test src/Gr4vy.Tests --filter "FullyQualifiedName~Gr4vy.Tests.Flows"
dotnet test src/Gr4vy.Tests --filter "FullyQualifiedName~TransactionLifecycleTest"
```

## How it is organised

```
src/Gr4vy.Tests/
  Utils/
    TestEnvironment.cs   # merchant-isolation harness + forward-compat interceptor + HTTP-call recorder
    Fixtures.cs          # mock-card inputs, addresses, cart items, unique ids
    Poll.cs              # bounded polling for eventually-consistent reads
    Reach.cs             # "endpoint reached" assertion for ops with no mock happy-path
  Flows/                 # headline stories: transaction + buyer lifecycle
  Processing/            # transactions, refunds, checkout sessions, payment methods, buyers, gift cards, digital wallets, payment links
  Backoffice/            # merchant accounts, payment services, 3DS, reports, payouts, audit logs, account updater
  PropertyTests.cs       # FsCheck property-based ("uncertainty") tests
  AuthTest.cs / WebhooksTest.cs / ...  # offline unit tests
```

### Isolation & parallelism

Each test **fixture** provisions its own merchant account (random id) with a
`mock-card` payment service, so fixtures share no state. They run concurrently —
both within a job (`[assembly: Parallelizable(ParallelScope.Fixtures)]` +
`LevelOfParallelism`, see `AssemblyInfo.cs`) and split across CI jobs.

### Forward compatibility

`TestEnvironment.JsonInterceptorHttpClient` injects a random unknown field into
every JSON response, asserting the SDK tolerates forward-compatible payloads.

### Property-based testing

`PropertyTests.cs` uses [FsCheck](https://fscheck.github.io/FsCheck/) — the .NET
analog of fast-check / hypothesis — with a bounded `MaxTest` and a fixed `Replay`
seed, so the generated cases are cheap and reproducible.

### Operations the sandbox can't fully exercise

Some operations need live external state (wallet provisioning, real network
tokens, payout-capable services). Those are exercised at the **request level**
via `Reach.ReachesAsync` — a real request is sent (so the endpoint is reached and
counts toward coverage) and a documented API error is accepted. These are always
explicit, never silently skipped.

## Endpoint coverage on PRs

CI runs the suite once with `GR4VY_TRACK_HTTP=1`, which records the method + path
of every request to `coverage/http/*.jsonl`. `scripts/endpoint-coverage.mjs`
matches those against the operation catalogue built from `src/Gr4vy/*.cs` and
posts a sticky PR comment listing endpoint reach and any **not-reached**
operations.

This matters for generated SDK-regen PRs: a new OpenAPI spec adds new operations
but no tests, so newly generated endpoints show up as "not reached" and tests can
be backfilled over time.

## CI

`.github/workflows/ci.yaml` shards the fixtures across a matrix, gates auto-merge
on a single stable `ci-complete` check (which only succeeds when every shard
passed), and posts the coverage comment in a separate, non-blocking job.
